# 当前改动说明

## 1. 这次改动和之前的区别

之前项目的后端本质上是：

- `pc-app/mock/server.js` 里的单文件 Express mock 服务
- 数据存储依赖 `pc-app/mock/db.json`
- 登录态依赖进程内存里的 `tokenStore`
- 更偏向课程演示，重点是“页面能跑”和“接口看起来像真的”

这次改动之后，后端变成了一个独立的最小可运行服务：

- 新增了独立目录 `backend/`
- 使用 `Express + TypeScript + Prisma + SQLite`
- 数据不再写入 `db.json`，而是写入 `SQLite` 数据库
- 登录态改成 `JWT`
- 保留了原有接口路径和主要字段名，尽量不改前端
- 目标不是做复杂业务，而是先把“真实后端基础链路”跑通

可以把这次改动理解成：

- 从“mock 演示后端”
- 升级为“真正可运行的第一阶段后端”

## 2. 当前已经实现的最小链路

当前后端已经跑通这些基础功能：

1. 验证码获取
2. 用户注册
3. 用户登录
4. 获取当前登录用户
5. 商家查看自己的酒店列表
6. 商家新增/编辑酒店
7. 管理员审核酒店状态
8. 商家给酒店新增房型
9. C 端公开查询已发布酒店列表
10. C 端公开查询酒店详情和房型

这意味着现在已经能支撑最小业务闭环：

1. 商家登录 PC 端
2. 新增酒店
3. 新增房型
4. 管理员审核通过
5. 小程序端看到已发布酒店

## 4. 用到了哪些后端操作

这次不是只写了接口，还实际用了下面这些典型后端操作。

### 4.1 服务启动与配置

- 使用 `Express` 启动 HTTP 服务
- 使用 `.env` 管理端口、数据库地址、JWT 密钥、验证码过期时间
- 默认监听 `3001`，兼容现有前端

### 4.2 数据建模

使用 `Prisma` 建了三张核心表：

- `User`
- `Hotel`
- `RoomType`

涉及的建模点：

- 主键自增
- 用户和酒店的一对多关系
- 酒店和房型的一对多关系
- 状态字段设计
- 时间字段 `createdAt / updatedAt`

### 4.3 数据库操作

使用了这些数据库操作：

- `findUnique`
- `findFirst`
- `findMany`
- `create`
- `update`
- `deleteMany`

对应能力包括：

- 按条件查用户
- 按角色过滤酒店
- 创建酒店
- 更新酒店状态
- 创建房型
- 联表查询酒店和房型

### 4.4 认证与鉴权

使用了：

- `bcrypt` 做密码加密和比对
- `jsonwebtoken` 签发和校验 JWT
- `Bearer Token` 鉴权

具体做了：

- 注册时对密码加密
- 登录时校验密码
- 登录成功后签发 JWT
- 受保护接口通过中间件校验 token
- 通过 `req.user` 传递当前用户

### 4.5 验证码

使用了：

- `svg-captcha`

具体做了：

- 生成 SVG 验证码
- 返回 `captchaId` 和 `captchaSvg`
- 在服务内缓存验证码文本和过期时间
- 登录/注册时校验验证码

### 4.6 权限控制

做了最基础的角色权限控制：

- 管理员可以审核所有酒店
- 商家只能查看和修改自己的酒店
- 商家只能给自己的酒店新增房型
- 公开接口只返回已发布酒店

### 4.7 数据格式兼容

为了不改现有前端，保留并兼容了这些字段名：

- `starLevel`
- `roomType`
- `openingDate`
- `merchantId`
- `status`
- `rejectReason`
- `updatedAt`

还兼容了房型创建入参：

- `roomTypeJson`
- `images`

### 4.8 种子数据

使用了 `Prisma Seed` 初始化测试数据：

- 管理员账号
- 商家账号
- 已发布酒店
- 待审核酒店
- 示例房型

这样做的作用是：

- 后端启动后可以直接联调
- 不需要手工先造数据

## 5. 当前相对之前的价值

这次改动最大的价值不是“多写了几个接口”，而是项目形态变了。

之前你能说的是：

- “我做了一个酒店管理和展示项目”

现在你可以更具体地说：

- “我把原本基于 mock 的项目，拆出了独立后端”
- “我用 TypeScript、Prisma 和 SQLite 搭了第一阶段真实后端”
- “我实现了注册登录、JWT 鉴权、角色权限控制、酒店审核与公开查询”
- “我在尽量不改前端的前提下完成了后端替换”

这在项目表达上会更像真实开发，而不是单纯页面项目。

## 6. 当前还没做的内容

为了先跑通基础能力，这一阶段刻意没做：

- 订单系统
- 库存系统
- 支付
- 消息通知
- 文件上传
- Redis
- MySQL
- 部署上线

这些会留到后续阶段。

## 7. 本地启动方式

### 启动后端

```powershell

Copy-Item .env.example .env
npm install
npm run db:push
npm run prisma:seed
npm run dev
```
