# 第一阶段剩余规划

## 1. 当前状态

第一阶段最小后端已经跑通，当前已经完成：

1. 独立 `backend/` 服务已建立
2. 技术栈已切到 `Express + TypeScript + Prisma + SQLite`
3. 已跑通认证、酒店管理、酒店审核、房型新增、公开酒店查询
4. 已用 `JWT` 替换原来 mock 服务里的内存 token
5. 已完成基础种子数据和本地联调

当前第一阶段剩下的工作，不再扩业务，而是把这套基础后端从“能跑”收敛到“能稳定联调、能交付说明、能继续迭代”。

## 2. 第一阶段剩余目标

第一阶段剩余目标只有 4 个：

1. 去掉前端里写死的后端地址
2. 关闭移动端默认回退本地 mock 的隐性逻辑
3. 补齐房型维护的最小闭环
4. 补齐启动、联调、验收文档

## 3. 明确不做的内容

这些内容不属于第一阶段剩余范围，暂时不做：

1. 订单系统
2. 库存系统
3. 价格日历
4. 支付
5. 消息通知
6. Redis
7. MySQL 切换
8. Docker 部署
9. 图片真实上传存储
10. 后台复杂报表

## 4. 剩余任务拆分

### 任务 A：前端接口地址配置化

目标：

- 不再把 `http://localhost:3001` 写死在前端代码里

要做的事：

1. PC 端把 `pc-app/src/services/authApi.js`
2. PC 端把 `pc-app/src/services/hotelApi.js`
3. PC 端把 `pc-app/src/services/roomTypeApi.js`
4. 小程序端把 `mobile-app/src/services/api.js`
5. 改成从环境变量或配置文件读取后端地址
6. 为本地开发保留默认值

预期结果：

1. 本机开发能继续访问 `3001`
2. 后续切局域网 IP 或远端地址时不用改业务代码

验收标准：

1. 前端代码里不再散落多个固定 `localhost:3001`
2. PC 端和小程序端都能通过配置切换接口地址

### 任务 B：移动端关闭默认 mock 回退

目标：

- 避免后端挂了但移动端还显示假数据，导致联调误判

要做的事：

1. 检查 `mobile-app/src/services/unifiedApi.js`
2. 把失败后自动回退 `shared/api.ts` 的逻辑改成显式开关
3. 默认联调时优先真实后端
4. 后端失败时给出明确错误或空状态

预期结果：

1. 后端有问题时，移动端能真实暴露问题
2. mock 数据只在明确开启时才生效

验收标准：

1. 后端停掉后，移动端不再静默显示本地假数据
2. 联调状态下，页面数据来源可确认是后端

### 任务 C：补房型编辑和删除

目标：

- 让商家后台具备最小的房型维护闭环

要做的事：

1. 后端新增 `PATCH /api/hotels/:hotelId/room-types/:roomTypeId`
2. 后端新增 `DELETE /api/hotels/:hotelId/room-types/:roomTypeId`
3. 增加商家权限校验
4. 增加基础参数校验
5. 如有必要，再决定是否补 PC 端最小入口

当前建议：

- 先把后端接口补上
- 前端入口是否补，取决于你这一阶段是否要把“维护闭环”展示完整

验收标准：

1. 商家只能修改自己的房型
2. 非法酒店或非法房型返回明确错误
3. 删除后公开详情页中该房型不再出现

### 任务 D：补文档和验收清单

目标：

- 让这套成果能被别人复现，不依赖口头说明

要做的事：

1. 补后端启动文档
2. 补前后端联调步骤
3. 补测试账号说明
4. 补阶段验收清单
5. 补接口总览文档

建议文档位置：

1. `codex改动/`
2. 或者根目录 README 追加“第一阶段后端联调”章节

验收标准：

1. 新人拿到仓库能按文档完成启动
2. 不需要口头解释就能完成基础联调

## 5. 推荐执行顺序

建议按这个顺序做，不要并行乱改：

1. 先做任务 A：前端地址配置化
2. 再做任务 B：移动端关闭默认 mock 回退
3. 再做任务 C：补房型编辑和删除接口
4. 最后做任务 D：补文档和验收清单

原因很简单：

1. 地址配置和 mock 回退会直接影响联调真实性
2. 房型编辑删除属于功能补齐，应该在联调环境可信之后再做
3. 文档必须以最终联调结果为准，适合最后整理
